# Verification Ladder — Verify, Don't Assume

작업을 **검증 가능한 목표(checkable goal)**로 바꾼 뒤, 충족될 때까지 루프를 돈다. 작업이 허용하는 한 **가장 높은 rung**을 사용한다.

## 검증 사다리 (높은 rung 우선)

1. **자동화 테스트 (rung-1)** — 변경 전 실패하고 변경 후 통과하는 테스트.
   (최선 — 정의 가능한 I/O가 있는 모든 작업에 사용. 버그 수정: 실패하는 테스트로 먼저 재현한 뒤 고친다.)
2. **타입 체크 + lint + 컴파일/빌드 (rung-2)** — clean하게 통과.
3. **실행 가능한 스모크 체크 (rung-3)** — 실제로 코드 경로를 실행해 출력을 보여주거나, 정확한 수동 재현 절차를 제시한다.
4. **명시적 수용 기준(plain words) + diff (rung-4)** — 위 세 가지가 적용 불가할 때(UI·디자인·탐색적 작업·재현 어려운 동시성/환경 버그). 사람을 검증자로 만든다 — 명시적으로.

## 두 가지 가드

- **Anti-overfit:** 테스트는 *구현*이 아니라 *요구사항*을 포착해야 한다. 통과한 뒤 자문하라 — "코드가 여전히 틀렸는데도 이게 통과할 수 있는가?" 그렇다면 체크가 불충분한 것이다.
- **Weak criteria:** 성공이 정의되지 않았다면, 구체적 기준 2~3개를 사용자에게 제안하고 명시한 기본값으로 진행한다 — 숨은 타깃을 지어내지 말고, 완벽한 스펙을 기다리며 멈추지도 말 것.

## 적용 원칙

- 작업을 시작하기 전에 **목표 / 제약 / 검증 방법**을 한 줄로 재진술한다(긴 세션에서 규칙이 흐려질 때 재정렬용).
- 가능한 한 낮은 rung으로 내려가지 말고, 작업이 허용하는 가장 강한 rung을 택한다. rung-4(사람=검증자)는 다른 rung이 모두 불가할 때의 최후 수단이다.
