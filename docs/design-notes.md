# 설계 결정 기록

직관에 반하거나 트레이드오프가 있는 결정을 남긴다. 코드가 "왜 이런지" 의 단일 출처.

## 1. 단일 에러 타입 + 단일 싱크

모든 실패를 `AppError(message, code)` 하나로 좁히고, 출력·종료는 `cli.ts` 한 곳에서만 한다.
명령은 try/catch 하지 않는다.

- **이유**: 에러 처리 분기가 흩어지지 않아 동작이 예측 가능하고, 종료 코드가 일관된다.
- **효과**: `--json` 모드에서 에러도 한 곳에서 구조화(`{ error: { message, code } }`)할 수 있다.

## 2. 모든 읽기/쓰기 명령이 3-모드

예외 없이 `--json`/`--quiet`/표를 제공한다(`core/output` 의 `render`/`reportWrite`).

- **이유**: 에이전트 자동화가 1순위 목표다. 쓰기 결과도 기계가 확인·체이닝할 수 있어야 한다.
- **참고 CLI 와의 차이**: 참고 구현은 여러 쓰기 명령이 텍스트만 냈다. 여기서는 쓰기도 `{ ...id, status }` 를 돌려준다.

## 3. `--json` 모드에서 에러도 구조화

실패도 `{ error: { message, code } }` 로 stderr 에 낸다.

- **이유**: 에이전트가 성공뿐 아니라 실패도 프로그램적으로 파싱·분기할 수 있어야 자동화가 견고하다.

## 4. `capabilities` 자기탐색 명령

전체 명령·인자·옵션을 구조화해 출력한다.

- **이유**: 에이전트가 사용 가능한 명령을 런타임에 발견하게 해, 스킬 문서가 낡아도 실제 능력을 알 수 있다.

## 5. 수정은 부분 수정 기본

`edit` 는 현재 값을 먼저 읽어, 지정하지 않은 필드를 유지한다.

- **이유**: "제목만 바꾸고 싶다" 가 흔한 요구다. 전체 페이로드를 매번 주게 하면 자동화가 번거롭고 본문을 실수로 비운다.

## 6. `post done` 은 closed 워크플로로 전이

전용 done 엔드포인트에 의존하지 않고, 워크플로 목록에서 `class === "closed"` 를 찾아 전이한다.

- **이유**: 서버가 별도 done 엔드포인트를 보장하지 않아도 동작하며, 워크플로 모델과 일관된다.

## 7. 파일 다운/업로드의 수동 307 처리

Dooray 파일 endpoint 는 307 로 실제 URL 을 Location 에 준다. ky 자동 follow 대신 `redirect:"manual"` 로 직접 처리한다.

- **이유**: 리다이렉트 대상에 인증 헤더를 다시 붙여야 하고, 307/308(메서드·바디 보존)만 처리해야 한다.
- **강건화**: 파일명은 content-disposition 에서 뽑되 `sanitizeFileName` 으로 경로 요소·`..` 를 제거한다(traversal 차단, 테스트 보유).

## 8. 이름 해석 + 파일 캐시

raw ID(15자리+)는 그대로 통과, 그 외는 목록을 받아 이름 매칭한다. 목록은 `~/.dooray-agent/cache` 에 TTL 로 캐시.

- **이유**: 사용자는 코드·이름으로 말하고, ID 는 모른다. 매 호출마다 목록을 다시 받지 않게 캐시한다.
- TTL: 자주 변하는 것(프로젝트·멤버) 1시간, 거의 불변(워크플로·위키) 24시간.

## 9. TS strict + `noUncheckedIndexedAccess`

배열·객체 인덱스 접근이 `T | undefined` 가 되도록 강제한다.

- **이유**: 페이지네이션·매칭에서 빈 결과를 놓치는 실수를 컴파일 단계에서 잡는다.
