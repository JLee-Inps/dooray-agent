Dooray 문서를 읽고 쓰는 CLI & 로컬 MCP 서버

NHN Dooray 의 위키·업무·캘린더·메일을 터미널에서 자동화한다. AI 에이전트가 결과를 파싱해 다음 행동으로 바로 잇도록, 모든 명령이 기계가 읽는 출력을 제공한다.

shell — 전역 설치 
$ npm install -g dooray-agent
3 실행 파일 · dra / dooray-agent / -mcp 27 MCP 도구 3-모드 출력 · json / quiet / 표

개요

무엇을 하나

사람이 터미널에서 쓰기에도 편하지만, 1순위 사용자는 AI 에이전트다. Dooray 문서를 조회·생성·수정·댓글까지 자동화하며, 두 가지 표면으로 노출된다.

CLI — dra
터미널·스크립트·Claude Code 에서 직접 실행. 모든 명령이 --json/--quiet/표 3-모드.
MCP 서버 — dooray-agent-mcp
Claude Desktop·Claude Code 가 도구로 호출하는 로컬 stdio 서버. 같은 패키지·같은 코어 공유.

토큰 한 번 등록하면 두 표면이 인증을 공유한다. 토큰은 로컬 밖으로 나가지 않는다.

시작하기

인증 & 첫 호출

Dooray 웹에서 개인 API 인증 토큰을 발급한 뒤 한 번만 등록한다.

토큰 발급 위치: Dooray 웹 → 우측 상단 프로필 → 개인 설정 → API(또는 "인증 토큰") 메뉴 → 토큰 생성 후 복사. 메뉴 이름은 버전에 따라 "API 토큰"·"OpenAPI" 등으로 다를 수 있다.
shell
$ dra login --token <DOORAY_API_TOKEN> --base-url https://api.dooray.com
$ dra whoami --json
{"id":"...","name":"...","externalEmailAddress":"..."}

토큰은 어디에 저장되나

입력한 토큰은 설치한 컴퓨터의 홈 디렉터리 한 파일에만 저장된다 — 다른 곳으로 동기화·업로드되지 않는다. 머신마다 각자의 파일을 갖는다.

~/.dooray-agent/config.json — 파일 권한 0600
{
  "token": "s1fv6l…",                 // Dooray API 토큰
  "baseUrl": "https://api.dooray.com"
  // 메일 사용 시에만: imapHost · smtpHost · mailUser · mailPassword
}
위치 · 권한
파일 ~/.dooray-agent/config.json 은 0600(본인만 읽기/쓰기), 디렉터리는 0700. 같은 컴퓨터의 다른 사용자도 못 읽는다.
로컬 전용
이 파일 밖으로 나가지 않는다. 환경변수 폴백 없음. 레포·npm 패키지·git 히스토리·로그 어디에도 없다.
유일한 전송처
Dooray API 요청 시 Authorization: dooray-api <토큰> 헤더(HTTPS)로만 나간다. 리다이렉트 시 같은 스코프에서만 재부착.
확인 · 삭제
dra config get 은 토큰을 s1fv**** 로 마스킹해 보여준다. rm ~/.dooray-agent/config.json 이면 완전 삭제(로그아웃 효과).

메일 설정 (선택 · IMAP/SMTP)

mail * 명령은 REST 토큰과 별개로 IMAP/SMTP 계정이 필요하다. 미설정이면 메일 명령만 code 4(설정 에러)로 설정 방법을 안내하고, 다른 명령은 정상 동작한다.

⚠ mail-password 는 로그인 비밀번호가 아니다. Dooray 계정 비밀번호가 아니라 IMAP/SMTP 전용 앱 비밀번호를 따로 발급해야 한다.
발급: Dooray 웹 → 개인 설정 → 메일 → IMAP/POP3(SMTP) 사용 활성화 → 화면에 나오는 앱 비밀번호 복사.
shell — 3가지 등록 후 조회
$ dra config set imap-host imap.dooray.com
$ dra config set mail-user <본인 Dooray 메일주소>
$ dra config set mail-password <IMAP 앱 비밀번호>   # 로그인 비번 아님!
$ dra mail list --limit 10 --json         # 등록 후 조회

메일 발송(SMTP)까지 쓰려면 dra config set smtp-host smtp.dooray.com 도 추가한다.

사용

기능 범위 — 되는 것 / 안 되는 것

지금 실제로 다룰 수 있는 것과, 아직 안 되는 것을 솔직하게 정리한다. 특히 캘린더는 일정 등록·조회·수정은 되지만 회의실·자원 예약은 안 된다 — Dooray 공개 API 가 자원 예약 전용 엔드포인트를 노출하지 않기 때문이다(제품 UI 에는 있으나 API 미지원).

캘린더
가능
  • 캘린더 목록 조회
  • 이벤트 목록 조회 (기본 지금 ~ +7일, 전체 캘린더)
  • 이벤트 단건 조회
  • 이벤트 등록 — 제목·시작·종료·본문(마크다운)·종일
  • 이벤트 수정 (부분 — 미지정 필드 보존) · 삭제
안 됨 · 제한
  • 자원 예약(회의실·자원) — 공개 API 미노출
  • 참석자 지정 입력 — 등록/수정 옵션 없음 (기존 참석자는 수정 시 보존만)
  • 반복 일정 · 장소 · 알림/리마인더
  • 넓은 기간 조회 — Dooray 최대 조회기간 제한(초과 시 에러)
업무
가능
  • 목록 · 조회 · 생성 · 수정(부분) · 검색
  • 완료 처리 · 워크플로 전이
  • 댓글 목록·추가·수정·삭제 (CLI)
  • 파일 첨부 · 다운로드
안 됨 · 제한
  • 수정 시 마일스톤·초기 워크플로 보존 안 됨 (알려진 한계)
  • MCP 도구는 댓글 목록·추가까지만 (수정·삭제는 CLI 전용)
  • 100건 초과 목록은 일부 누락 가능 (알려진 한계)
위키
가능
  • 페이지 목록 · 조회
  • 생성 (부모 미지정 시 루트 하위 자동) · 수정(부분) · 삭제
  • 댓글 목록·추가·수정·삭제 (CLI)
  • 파일 첨부 · 다운로드
안 됨 · 제한
  • MCP 도구는 댓글 추가만 (목록·수정·삭제는 CLI 전용)
  • 페이지 이동(부모 변경) 미노출
메일 IMAP/SMTP 옵트인
가능
  • 최근 메일 목록 조회 (IMAP)
  • 단건 조회 — 본문 파싱 (IMAP)
  • 발송 — 받는사람·제목·본문 (SMTP)
안 됨 · 제한
  • 참조(cc/bcc) · 첨부 입력 없음
  • 메일 검색 없음
  • REST 토큰과 별개로 IMAP/SMTP 설정 필요
전반 한계. messenger·drive 는 실험적(공개 API 미검증 — 경로·응답 형태가 실제와 다를 수 있음). 검색은 키워드 기반이며 문서 전체를 관통하는 시맨틱/전문 검색은 없다. 표면(CLI vs MCP)에 따라 노출 범위가 다른 부분이 있으니 위 표를 기준으로 삼는다.

출력 모드 & 종료 코드

모든 읽기/쓰기 명령이 세 가지 모드를 제공한다 — 예외 없음.

모드용도

--json구조화된 JSON — 에이전트는 항상 이걸 쓴다. 읽기는 API 원자료, 쓰기는 { ...id, status }.
--quiet식별자만 — 파이프용.

실패도 파싱 가능하다 — --json 에러는 { error: { message, code } }. 종료 코드로 원인을 구분한다:

0 성공 1 API 2 인증 3 사용법 4 설정

아키텍처

단방향 데이터 흐름. 각 계층은 아래 계층만 알고, 위를 모른다. CLI 와 MCP 는 병렬 진입점이며 하위 계층(이름해석·API·코어)을 공유한다.

CLI 경로
cli.ts
진입점 · 전역 옵션 · 단일 에러 싱크
→
commands/*
얇은 오케스트레이터
→
resolve/* · client
이름→ID · Dooray API
→
core/output
3-모드 렌더
MCP 경로 — 하위 계층 공유
mcp.ts
stdio 진입
→
mcp/serve
도구 등록 · 에러 매핑
→
mcp/tools
27 핸들러 → resolve/* · client
src/cli.ts
진입점. 전역 옵션(json/quiet/no-color)과 유일한 에러 출력·종료 지점. 명령은 try/catch 하지 않고 여기로 버블한다.
src/core/
횡단 인프라 — errors(AppError+ExitCode) · output(3-모드) · config · cache(TTL) · session.
src/dooray/
API 계층 — ky 래퍼, 단일 에러 정규화, 페이지네이션. mail 은 IMAP/SMTP.
src/resolve/
사용자 문자열 → Dooray ID. 캐시 + 이름 매칭(정확 → 부분 → 모호 에러).
src/commands/
명령 그룹. 각 액션은 읽기·해석·쓰기·렌더를 잇는 얇은 오케스트레이터.
src/mcp/
두 번째 진입 표면. serve(등록·에러 매핑) + tools(27 핸들러)가 resolve/client 를 재사용.

구조

설계 원칙

01
단일 에러 타입 · 단일 싱크
AppError(message, code) 하나. 명령은 삼키지 않고 진입점으로 버블하며, 출력·종료는 한 곳에서만 일어난다.
02
3-모드 출력, 예외 없음
모든 읽기/쓰기가 json/quiet/표를 제공. 데이터는 stdout, 스피너·에러는 stderr — 기계 출력이 오염되지 않는다.
03
부분 수정 기본
edit 는 지정 안 한 필드를 현재 값으로 유지한다. 서버가 전체 교체(PUT)라도 현재값을 읽어 재공급한다.
04
이름 해석 + 캐시
프로젝트 코드·워크플로 이름을 ID 로 해석하고 파일 캐시(TTL)로 반복 호출을 줄인다. raw ID 는 통과.
05
계약이 단일 출처
contracts/ 가 명령 표면·타입·MCP 도구의 불변 SSOT. 구현은 이를 따르고 타입 검사가 드리프트를 잡는다.
06
TS strict · ESM · named export
noUncheckedIndexedAccess 포함 strict, ESM 모듈, default export 금지. tsup 로 dual-bin 번들.
